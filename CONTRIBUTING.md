# Contributing

Thanks for your interest in contributing to this SDK! This document provides guidelines for contributing to the project.

## Getting Started

### Prerequisites

- Go 1.21+
- Docker (only required to run wire tests; see [Wire Tests](#wire-tests))

### Installation

Install the project dependencies:

```bash
go mod tidy
```

### Building

Build the project:

```bash
go build ./...
```

### Testing

Run the test suite:

```bash
go test ./...
```

#### Wire Tests

If this SDK includes a `wiremock/` directory, the test suite contains wire tests that exercise the client against a [WireMock](https://wiremock.org/) server. The `go test` command above expects `WIREMOCK_URL` to point at a running WireMock instance.

Start WireMock, run the tests, then tear it down:

```bash
docker compose -f wiremock/docker-compose.test.yml up -d
export WIREMOCK_URL=http://localhost:$(docker compose -f wiremock/docker-compose.test.yml port wiremock 8080 | cut -d: -f2)
go test ./...
docker compose -f wiremock/docker-compose.test.yml down
```

The compose file maps WireMock's container port `8080` to a random host port, which is why `WIREMOCK_URL` is derived from `docker compose ... port` rather than hardcoded.

### Formatting

Format code:

```bash
gofmt -w .
```

Or equivalently:

```bash
go fmt ./...
```

### Vetting

Run the Go vet tool to catch common mistakes:

```bash
go vet ./...
```

## About Generated Code

**Important**: most of this SDK is generated by [Fern](https://buildwithfern.com) from
`specs/production/openapi.json`. Edits to generated files are overwritten on the next
regen. Generated files carry a `// Code generated by Fern. DO NOT EDIT.` header; the
`core/` and `internal/` trees and the root support files (`pointer.go`, `file_param.go`)
are also generated despite not all carrying the marker.

Hand-written, and safe to edit:

| Path | What it is |
|---|---|
| `drivers/` | The Go MobileDriver (phone control over the DCP WebSocket) |
| `scripts/` | The regen script |
| `fern/`, `specs/` | Fern config and the source spec |
| `.github/` | CI and the regen/release pipeline |
| `VERSION` | The release version (see Releasing, below) |
| `CONTRIBUTING.md` | This file |

### How regen actually preserves hand-written code

**`.fernignore` does not work in this repo. Do not use it.**

`fern-go-sdk`'s `--local` generation wipes every non-generated file in its output
directory — verified: it deletes `drivers/`, `.github/` and `VERSION` **even when they
are `.fernignore`-listed**. Fern's [custom-code docs](https://buildwithfern.com/learn/sdks/overview/custom-code)
describe a mechanism this generator does not honour in local mode.

So `scripts/regen.sh` generates into a throwaway `.gen/` and rsyncs the result up to
the repo root:

```
fern generate --local  ->  .gen/          # .fernignore is a no-op here
rsync -a --delete .gen/ ./ --exclude=...  # the exclude list is the real protection
```

**The rsync exclude list in `scripts/regen.sh` is the only thing protecting
hand-written code.** `--delete` prunes anything at the root that isn't in the generated
output and isn't excluded.

### Adding a hand-written path

To add hand-written code that survives regen:

1. Add the path to the rsync `--exclude` list in `scripts/regen.sh`.
2. If it needs to be committed by the regen PR, add it to `add-paths` in
   `.github/workflows/regen.yml`.
3. If it pulls in dependencies the generated code doesn't import, confirm they survive
   `go mod tidy` — regen re-tidies against the full tree for exactly this reason
   (`drivers/mobile` pulls in `coder/websocket`, which the generator would otherwise drop).

Skip step 1 and your code is deleted on the next regen, silently, inside an
auto-opened PR. Adding it to `.fernignore` will not save you.

### Changing how code is generated

The Go SDK generator lives in the [Fern repository](https://github.com/fern-api/fern)
under `generators/go-v2/`; see the
[Fern contributing guidelines](https://github.com/fern-api/fern/blob/main/CONTRIBUTING.md).
Best for bug fixes in generated code and improvements that benefit all users.

## Releasing

Regen never releases. A regen PR updates the generated client and the spec, and
deliberately does not touch `VERSION`.

To cut a release, edit `VERSION` and merge to `main`. `.github/workflows/tag-on-merge.yml`
picks that up, creates the annotated `vX.Y.Z` tag and the GitHub release, then dispatches
to `axilioai/cli` to open a dependency-bump PR. For a Go module the tag *is* the publish —
there is no registry step.

## Making Changes

### Workflow

1. Create a new branch for your changes
2. Make your modifications
3. Run tests to ensure nothing breaks: `go test ./...`
4. Format your code: `gofmt -w .`
5. Vet your code: `go vet ./...`
6. Build the project: `go build ./...`
7. Commit your changes with a clear commit message
8. Push your branch and create a pull request

### Commit Messages

Write clear, descriptive commit messages that explain what changed and why.

### Code Style

This project uses `gofmt` for code formatting. Run `gofmt -w .` before committing to ensure your code meets the project's style guidelines.

## Questions or Issues?

If you have questions or run into issues:

1. Check the [Fern documentation](https://buildwithfern.com)
2. Search existing [GitHub issues](https://github.com/fern-api/fern/issues)
3. Open a new issue if your question hasn't been addressed
